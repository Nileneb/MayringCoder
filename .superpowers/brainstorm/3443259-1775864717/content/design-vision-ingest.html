Design: Vision Memory Ingest

Repo-Bilder → qwen2.5vl Caption → ChromaDB → Pi search_memory

Pipeline Flow

checker.py --ingest-images <repo>
↓
git clone --depth=1 → /tmp/mayring-<slug>/
↓
find *.png *.jpg *.svg *.gif *.webp (skip >5MB, max 50 Bilder)
↓ pro Bild:
vision_captioner.caption_image(path, model="qwen2.5vl:3b")
↓
memory_ingest.ingest(source_type="repo_file", category="visual")
↓
ChromaDB + SQLite → Pi findet Captions via search_memory()
↓
rm -rf /tmp/mayring-<slug>/

Neue / geänderte Dateien

Datei Aktion Was
src/image_ingest.py NEU clone, find, caption, ingest — Haupt-Logik
src/vision_captioner.py minimal bereits fertig, ggf. Timeout-Parameter ergänzen
checker.py +Flag --ingest-images analog zu --ingest-issues
src/memory_ingest.py unverändert source_type="repo_file" bereits unterstützt

Chunk-Format im Memory

# was Pi bei search_memory("datenbankstruktur") findet:
{
  "source_type": "repo_file",
  "category": "visual",
  "filename": "docs/architecture.png",
  "content": "[Caption] Tasks-Tabelle hat user_id → users.id FK. Projects haben budget_used und budget_total Spalten...",
  "repo": "nileneb-applinngames"
}

Limits & Guards

  • Max Bildgröße: 5 MB (skip + warn)
  • Max Bilder pro Run: 50 (konfigurierbar via --max-images N)
  • VISION_MODEL env var: default qwen2.5vl:3b
  • Dedup: Hash-Check via memory_store — gleiche Datei wird nicht neu gecaptioned
  • Cleanup: rm -rf /tmp/mayring-slug/ nach Ingest immer, auch bei Fehler
