Visual Qwen Pipeline — Ideen-Überblick

Wo könnte Vision in MayringCoder reinpassen? 3 Richtungen zur Auswahl.

Aktueller Stand (Text-only)

GitHub Repo → gitingest → Splitter / Categorizer → LLM (Qwen Text) → JSON Findings → Report.md
vision_captioner.py existiert bereits, ist aber noch nicht in die Pipeline integriert

Wähle die Richtung, die dich interessiert:

A

Repo-Bilder analysieren

qwen2.5vl:3b captioniert PNG/SVG/JPG-Dateien im Repo (ERDs, Architecture-Diagramme, Screenshots). Caption fließt als Kontext in die normale Text-Analyse ein. Vision als Pre-Processing Schritt.

Pro

  • vision_captioner.py fast fertig
  • Minimal-Eingriff in Pipeline
  • Liefert echten Mehrwert bei Bild-reichen Repos

Con

  • Braucht qwen2.5vl:3b zusätzlich zu laufen
  • VRAM-Konkurrenz mit Hauptmodell
B

Screenshot → Code Review

User schickt Screenshots von UI, Fehlermeldungen oder Logs. Fine-tuned Pi + mmproj.gguf analysiert direkt: "Was stimmt hier nicht?" — multimodales Code-Review ohne Repo-Zugriff.

Pro

  • mmproj.gguf bereits generiert
  • Einzigartiger Use-Case
  • Kein Repo-Fetch nötig

Con

  • Qwen3-2B-Base ist kein Vision-Modell — mmproj evtl. nutzlos
  • Braucht llama-mtmd-cli statt Ollama
  • Unclear ob das fine-tuned model vision kann
C

Visuelles Pipeline-Dashboard

web_ui.py (existiert bereits!) um eine Live-Visualisierung erweitern: Repo-Graph, Findings-Heatmap, Pi-Agent Tool-Call-Trace, Memory-Stats. Kein neues Vision-Modell — visuelle Ausgabe statt visueller Eingabe.

Pro

  • web_ui.py als Basis vorhanden
  • Kein extra GPU-RAM
  • Macht Ergebnisse sofort erklärbar

Con

  • Nicht "Vision" im AI-Sinne
  • Mehr Frontend-Arbeit
